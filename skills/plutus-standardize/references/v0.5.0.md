# plutus-verify 0.5.0 notes

Deltas from 0.4.x. When `plutus_verify.__version__` reports `0.5.0` or higher, apply
these constraints. (0.4.4–0.4.6 were unreleased and folded into 0.5.0.)

> Source: [CHANGELOG.md `[0.5.0]`](../../../CHANGELOG.md).

## Changed — `check` is read-only; produced output goes to a harvest buffer

`plutus check` no longer writes the working tree. Each step's declared `step.outputs`
are harvested to the **gitignored `.plutus/results/<step>/`** and compared there;
the committed `result/…` files are never overwritten. Earlier steps reach later ones
through that buffer (the inter-step data bus), so multi-step pipelines reproduce
without touching cwd.

- **Phase 3 step 5b (path A chart baseline) — the overwrite race is gone.** Still
  copy the README-referenced chart into `.plutus/expected/<step>/<path>` for path A
  (that's the verification baseline), but the old worry that a later `check`/smoke-run
  would overwrite the committed `result/` chart no longer applies — `check` is
  read-only and produced charts land in `.plutus/results/`.

## Changed — `snapshot` runs in-container by default

`plutus snapshot .` builds the image and runs every step in the container (the same
env `check` reproduces), harvests to `.plutus/results/`, then blesses: metric numbers
→ `manifest.yaml`, artifact files → `.plutus/expected/`, **plus a human-facing copy
→ `result/`**. This is the path-B baseline capture (script-is-truth). `--no-run` stays
as the local-bytes opt-out (bless pre-existing local outputs).

## Added — data download cache (`.plutus/cache/`)

Data sources fetch into the **gitignored `.plutus/cache/`** (persisted across runs;
clear it to refetch), never the working tree, and overlay into each step's sandbox.
`gdown` now ships in the `runner` extra, so `google_drive` sources work out of the
box (no more `ModuleNotFoundError: gdown`).

## Added — `env.install_project` (uv-only)

For installable-package repos whose steps invoke console scripts (e.g.
`pmm-backtest`) or `python -m <pkg>`, set `env.install_project: true` so the image
installs the repo's own package. Requires `manager: uv` + a committed lockfile + a
`pyproject.toml` at the repo root. Default false. See D6 in
[`decision-tree.md`](decision-tree.md).

## Changed — `plutus init` scaffolds `.dockerignore`

`plutus init` now writes the `.dockerignore` baseline at setup time (commit it). The
read-only `check`/`snapshot` no longer surprise-write it; the build-time write remains
only as a safety net.

## Skill behavior on 0.5.0

- **Gitignore (Phase 3 step 7) must include `.plutus/results/` and `.plutus/cache/`**
  in addition to `.plutus/run/` and `.plutus/build/`. Without them, `check` runs and
  data fetches dirty the tree — the opposite of the read-only design.
- **Verify (Phase 4):** after `plutus check .`, the working tree stays clean — a
  dirty tree is now a real smell (a missing gitignore entry), not expected chart
  churn. Produced artifacts to inspect on a mismatch are under
  `.plutus/results/<step>/` (vs the committed `.plutus/expected/<step>/`).
- **install_project repos:** a clean `plutus check` exit 0 means the package built and
  its console scripts ran in-container.

## Backward compatibility

Existing 0.4.x-era manifests work without changes. `env.install_project` defaults to
`false`. No manifest-schema breakage; the only new field is the optional
`env.install_project`.
