# plutus-verify 0.2.6 notes

Deltas from 0.2.5. When `plutus_verify.__version__` reports `0.2.6`, apply these constraints.

> Source: [docs/others/zbounce-v1-to-v2-upgrade.md](../../../docs/others/zbounce-v1-to-v2-upgrade.md) §6.

## Breaking — Python API only

No manifest, CLI, or SDK changes.

- `plutus_verify.spec.runtime.refcompare` module renamed to `plutus_verify.spec.runtime.artifact_compare`.
- `compare_artifact` and `CompareResult` are re-exported from the new module.
- `V2RuntimeResult.reference_results` (already-renamed-in-0.2.5) is now consistently `.artifact_results` everywhere.

## Fixed — the load-bearing fixes for this Skill

- The missing-snapshot strict gate in `compare_artifact` moved into per-comparator logic. **`visual_similarity` treats missing expected-path as a non-blocking SKIP**, symmetric to the missing-vision-client SKIP. `byte_exact` and `json_numeric_tolerance` keep the strict gate (correctly — they're deterministic comparators where missing reference = real failure).
- `plutus check` report now renders **per-step artifact lines**, e.g.:
  ```
  SKIP visual_similarity result/backtest/hpr.svg  [skipped (no vision client configured; pass --visual-check to enable)]
  ```
  Closes the silent-fail UX gap from 0.2.0 / 0.2.5 (gotcha **G4**).

## Added

- `CompareResult.skipped: bool` — distinguishes "verified pass" from "not verified" honestly.
- `CompareResult.path: str` — stamped by the dispatcher for the renderer.

## Skill behavior on 0.2.6

- Phase 3 step 5b chart baseline: same as 0.2.5 — copy README-referenced charts into `.plutus/expected/<step>/<path>` during manifest authoring (D4 = path A only). The Phase 3 step 5b mechanism is now *more* important on 0.2.6 than on 0.2.5: a missing baseline used to hard-fail (0.2.5), but on 0.2.6+ it silently SKIPs. Without the Phase 3 step 5b copy, the maintainer would get a green `plutus check` but the `visual_similarity` checks would all be non-verifying SKIPs. The Skill's job is to set up *real* verification, so this step stays required.
- Phase 4 verification: **do NOT run `plutus snapshot`** for D4 = path A. See **G7** in [`known-gotchas.md`](known-gotchas.md). For D4 = path B, snapshot in Phase 4 step 2 is correct.
- Phase 4 diagnosis: artifact-level results are now rendered in the report; failures are visible without manual repro. Step stderr surfacing is still deferred — manual `docker run` repro from **G5** remains the diagnostic path for step crashes.
- Phase 3 manifest: identical shape to 0.2.5. No migration needed.
