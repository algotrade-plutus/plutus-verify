# plutus-verify 0.5.0 — scoring nuances

Deltas from 0.2.9 that affect how the rubric is applied. When
`plutus_verify.__version__` reports `0.5.0` or higher, apply these constraints.

> Source: [CHANGELOG.md `[0.5.0]`](../../../CHANGELOG.md).

## Reproducible bucket (50) — a clean tree after `check` is now the norm

On 0.5.0 `plutus check` is **read-only**: produced artifacts are harvested to the
gitignored `.plutus/results/<step>/` (compared there, never overwriting `result/`),
and data sources download into the gitignored `.plutus/cache/`. Implications for
scoring the "were any tracked config files modified / is the tree dirty after
`check`?" signal:

- **A dirty working tree after `check` is now a real smell**, not expected chart
  churn — it means the repo is **missing gitignore entries** (`.plutus/results/`
  and/or `.plutus/cache/`) or a step writes outside its declared `outputs`. Dock the
  Tidy bucket and flag it as an improvement path (add the gitignore lines).
- The pre-0.2.10 "runtime mount bypasses `.dockerignore`" caveat is gone; a passing
  `check` on 0.5.0 reflects a genuine isolated build + run.
- Baseline provenance is cleaner: groundtruth lives only in `.plutus/expected/` +
  the manifest `expected.metrics[].value`, written solely by `plutus snapshot`.

## `env.install_project` repos

If the manifest sets `env.install_project: true`, a passing `check` means the repo's
**own package built and its console scripts ran in-container** — strong evidence for
the Reproducible bucket (the pipeline is invoked exactly as a user would, via the
installed entry points). Confirm `manager: uv` + a committed lockfile + a root
`pyproject.toml` are present (the validator requires them); their absence with
`install_project: true` is a hard manifest error, not a passing repo.

## Standardized / template bucket (10)

0.5.0 repos are expected to gitignore all four ephemeral dirs (`.plutus/run/`,
`.plutus/results/`, `.plutus/cache/`, `.plutus/build/`) and to commit the
`.dockerignore` that `plutus init` scaffolds. A standard, `plutus-document`-rendered
README (metric tables from `expected.metrics`, chart embeds, the pinned-wheel repro
block) is full credit here.

## What didn't change

- The 50/25/10/15 bucket weights and the read-only, non-modifying nature of the
  scoring skill itself.
- Metric/artifact comparison semantics (tolerance, `byte_exact` / `json_numeric` /
  `visual_similarity`).
