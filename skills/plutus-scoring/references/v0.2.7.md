# plutus-verify 0.2.7 — scoring nuances

Deltas from 0.2.6 that affect how the rubric is applied. When `plutus_verify.__version__` reports `0.2.7` or higher, apply these constraints.

> Source: [docs/completion-report/2026-05-27-v0.2.7-byte-fallback-and-skill-split.md](../../../docs/completion-report/2026-05-27-v0.2.7-byte-fallback-and-skill-split.md).

## Reproducible bucket — WARN counts as a pass

0.2.7 introduced a fourth `CompareResult` state: `WARN` (`ok=False`, `skipped=True`), emitted by the byte-comparison fallback when `visual_similarity` is declared but no `vision_client` is configured AND bytes differ between produced and expected files.

For scoring purposes:

- **`ok` lines** — verified passes. Reproducible at full credit.
- **`SKIP` lines** — non-verified but not problematic (missing reference, missing LLM). Reproducible at full credit; the rubric does not penalize for absent baselines.
- **`WARN` lines** — divergence detected (bytes differ) but inconclusive without an LLM. Reproducible at full credit *for now* — the exit code is 0, and the rubric is anchored on "`plutus check` exits 0." But surface in Phase 2 improvement paths: "Consider `--visual-check` to definitively verify the chart divergence."
- **`FAIL` lines** — verified divergence. Reproducible takes the hit.

## Output kind labels

The `plutus check` report distinguishes `kind` in artifact lines:

- `kind=visual_similarity` → LLM judged (passed/failed) or missing reference (SKIP).
- `kind=byte_identical` → byte fallback ran (ok/WARN). Internal label; user can't declare it in the manifest.
- `kind=byte_exact` → user-declared strict byte check (for deterministic files like CSVs).
- `kind=json_numeric_tolerance` → JSON-structural check.

Use the kind to inform the Tidy/Standardized buckets — e.g., if a chart artifact reports `WARN byte_identical` consistently across runs, the script likely has embedded timestamps or font-rendering jitter. Mention that as a polish improvement (Tidy +0 to +2) rather than a true reproducibility issue.

## Phase 3 re-run command — no flag changes

The re-run command emission stays the same as 0.2.6. `--visual-check` is still the user-opt-in flag for LLM-based comparisons; without it, the byte fallback runs and surfaces SKIP/WARN/ok appropriately.
