# plutus-verify 0.2.9 — scoring nuances

Deltas from 0.2.8 that affect how the rubric is applied. When `plutus_verify.__version__` reports `0.2.9` or higher, apply these constraints.

> Source: [CHANGELOG.md `[0.2.9]`](../../../CHANGELOG.md).

## Auto-emits `.dockerignore` if absent; manifest-declared secret routing now authoritative for bridge steps

Pre-0.2.9, a bridge step could pass `plutus check` despite no real environment-variable forwarding because the container read a leaked `.env` directly. That made the "Reproducible" bucket look better than it actually was — a step could exit 0 by bypassing the declared posture.

0.2.9 closes that hole by seeding `<repo>/.dockerignore` with `.env*` and friends excluded.

## Skill behavior on 0.2.9

- **Reproducible bucket** — when assessing a bridge step's exit-0 status, the score reflects real verification of the **image build context** on 0.2.9+. The runtime volume mount still bypasses `.dockerignore` (see "Caveat" below) — score Tier 3 bridge steps conservatively. Pre-0.2.9 repos might have been over-credited; rescore if a maintainer reports verifying with an older version.
- **Tidy bucket** — check whether the repo has a `.dockerignore`:
  - Auto-generated baseline present (has the `# Auto-generated by plutus-verify` header) → no penalty; baseline is the safe default.
  - User-authored `.dockerignore` with tier-appropriate exclusions → small bonus toward Tidy (concrete maintenance care visible in the diff).
  - No `.dockerignore` at all (running pre-0.2.9 or somehow deleted) → flag in Phase 2 improvement paths: "Add a `.dockerignore` (or upgrade to plutus-verify ≥0.2.9 which auto-emits one) to prevent secret-leak into Docker image layers."
- **Standardized bucket** — orthogonal; no change.

## Caveat — bridge step "ok" is conditional on host cleanliness

0.2.9 closes the **image-layer** leak (`docker pull` consumers can no longer recover `.env` from layers) but the **runtime volume mount** is unchanged — `runner_docker.py` still does `-v {cwd}:/srv/repo`, which exposes host `.env` and host `data/cache/*.parquet` to the container regardless of `.dockerignore`.

When scoring Reproducible on a Tier 3 repo:

1. Inspect host environment at score time: `env | grep '^DB_'` should show the keys the manifest declares in `secrets:`. Empty output + passing `data_collection` step = scored "ok" was probably a `.env`-via-mount leak, not real `--secrets-from-env` routing.
2. Inspect cache mtimes across two consecutive `plutus check` runs:
   ```bash
   stat -f '%Sm' data/cache/<symbol>/1min/<month>.parquet
   ```
   Unchanged mtime = bridge short-circuited via the mount, not via DB.

If either signal fires, flag in Phase 2 improvement paths: "Tier 3 bridge step's verification depends on host cleanliness (no host `.env`, no stale `data/cache/`). Add a per-run cache wipe and require explicit env-var export for repeatability." See [G11 in the transform skill's known-gotchas](../../plutus-transform/references/known-gotchas.md#g11) for the recipe.
