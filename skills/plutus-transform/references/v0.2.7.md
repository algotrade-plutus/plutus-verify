# plutus-verify 0.2.7 notes

Deltas from 0.2.6. When `plutus_verify.__version__` reports `0.2.7` or higher, apply these constraints.

> Source: [docs/completion-report/2026-05-27-v0.2.7-byte-fallback-and-skill-split.md](../../../docs/completion-report/2026-05-27-v0.2.7-byte-fallback-and-skill-split.md).

## Behavior change — `visual_similarity` falls back to byte comparison

In 0.2.6, `visual_similarity` with no vision client returned a uniform SKIP. 0.2.7 byte-compares as a fallback:

| Files | 0.2.6 result | 0.2.7 result |
|---|---|---|
| byte-identical | `SKIP visual_similarity [no vision client]` | `ok byte_identical [bytes match (no LLM check needed)]` |
| byte-different | `SKIP visual_similarity [no vision client]` | `WARN byte_identical [bytes differ; pass --visual-check for LLM judgment]` |

`byte_identical` is an internal `kind` label — never declarable in the manifest. The schema still accepts only `json_numeric_tolerance`, `visual_similarity`, `byte_exact` as `compare:` values.

## Added — 4-state CompareResult matrix

`(ok, skipped)` field combinations:

| `ok` | `skipped` | meaning | marker | exit-code |
|---|---|---|---|---|
| T | F | verified pass | `ok` | 0 |
| T | T | not verified, no evidence of issue | `SKIP` | 0 |
| F | T | divergence detected but inconclusive (new) | `WARN` | 0 |
| F | F | verified divergence | `FAIL` | 1 |

Exit-code logic updated: `skipped=True` is non-blocking regardless of `ok`. Previously the check was `if not r.ok: return 1`; now it's `if not r.ok and not r.skipped: return 1`.

## Skill behavior on 0.2.7

- Phase 3 step 5b chart baseline: same as 0.2.5 / 0.2.6 (copy README-referenced charts into `.plutus/expected/` during manifest authoring, D4 = path A only). The Phase 3 step 5b copy is what makes `byte_identical` matches possible at `plutus check` time. Without it, every chart in a no-LLM run would WARN (bytes-differ-from-nothing → falls back to missing-expected SKIP, which is fine but less informative).
- Phase 4 verification: expect a mix of `ok byte_identical`, `WARN byte_identical`, and possibly `SKIP visual_similarity` lines depending on chart determinism. All are non-blocking. The maintainer can opt into `--visual-check` to upgrade WARN/SKIP to real LLM judgment.
- Phase 4.5 transform summary: `WARN byte_identical` lines are *not* worked-around smells — they're informational. Don't surface them as architectural issues. Real chart non-determinism (matplotlib timestamps, font availability) is a polish concern at most.
- Phase 6 divergence: a `WARN` line appearing where the operator expected `ok` is *not* a divergence — it's the framework's normal behavior. Only trigger Phase 6's substantive report if the WARN/SKIP/ok distinction itself caused 3+ rounds of operator confusion.

## Forward compatibility

The `byte_identical` kind label may be extended in future versions if the framework adds smarter byte-pre-filters (e.g., timestamp-stripped equality). The Skill should pattern-match on the `kind` field literally; don't assume the set is closed.
