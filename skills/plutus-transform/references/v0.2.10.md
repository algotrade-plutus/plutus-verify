# plutus-verify 0.2.10 notes

Deltas from 0.2.9. When `plutus_verify.__version__` reports `0.2.10`
or higher, apply these constraints.

> Source: [CHANGELOG.md `[0.2.10]`](../../../CHANGELOG.md).

## Changed ŌĆö per-step staging replaces direct mount

Each step now runs in a tempdir populated from cwd through the
`.dockerignore` filter (plus `step.inputs` if declared). The container
never sees the maintainer's host tree directly. Closure of the
runtime-mount leak [G11](known-gotchas.md#g11) documented for 0.2.9.

## Skill behavior on 0.2.10

- **Phase 3 step 6 (`step.outputs`) is now load-bearing.** Anything a
  step writes outside `step.outputs` is discarded at the stagingŌåÆcwd
  extraction step. Audit each step: what does the script write?
  Declare it.
  - For `data_collection`: usually `data/cache/`.
  - For `in_sample_backtest`: usually `result/backtest/`.
  - For `optimization`: usually `parameter/`, `result/optimization/`.
  - The framework always returns `.plutus/run/<step>/` ŌĆö never
    declare that one yourself.
- **Phase 3 step 6.5 (new) ŌĆö `step.inputs` is a complete-coverage
  allowlist, not a data-inputs declaration.** When non-empty, `inputs`
  becomes a positive filter on top of `.dockerignore`: ONLY paths
  matching `inputs` patterns get copied into the step's staging
  directory. That includes:
  - The script binary itself (e.g. `scripts/plutus_emit_*.py`)
  - All source dirs the script imports from (`src/`, `config/`)
  - Any reference data files the script reads
  - `pyproject.toml` / `setup.py` if `pip install .`-installed package
    metadata isn't enough (rare)

  Most Tier 3 `data_collection` steps need at least
  `[src/, config/, pyproject.toml]` plus whatever data paths the
  loader reads. Wrapper steps need `[scripts/<wrapper>.py,
  reports/<run>/]` plus the SDK's dependencies (handled by the
  framework via the image).

  **Default for new manifests until you've audited each step's file
  surface: `inputs: []`** ŌĆö empty falls back to the v0.2.9-style
  `.dockerignore`-only filter, which is safe by default. Tighten
  step-by-step once you've confirmed the manifest works. See
  [G12 in known-gotchas](known-gotchas.md#g12) for the failure mode if you
  mis-declare.
- **Phase 4 diagnosis.** A passing-and-real `data_collection` on
  0.2.10 means the bridge actually queried the DB ŌĆö no other path
  can succeed. The G11 mtime-and-host-env recipe is no longer needed
  for 0.2.10+ but stays as a diagnostic for "is this repo running on
  an older framework?"

## Backward compatibility

Existing 0.2.9-era manifests work without changes. Scripts that wrote
files outside any declared output will lose those files ŌĆö add them to
`step.outputs` if needed. No CLI / SDK / manifest-schema breakage.

## What didn't change

- `.dockerignore` auto-emission (still happens, still preserves
  user-authored).
- Skill phases 1-2 (Survey/Decide).
- Manifest schema (`steps[].inputs` and `steps[].outputs` already
  existed ŌĆö 0.2.10 just makes them load-bearing at runtime).
- `--secrets-from-env` routing (now actually authoritative, no longer
  advisory on Tier 3).
