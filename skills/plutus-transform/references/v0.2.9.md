# plutus-verify 0.2.9 notes

Deltas from 0.2.8. When `plutus_verify.__version__` reports `0.2.9` or higher, apply these constraints.

> Source: [CHANGELOG.md `[0.2.9]`](../../../CHANGELOG.md).

## Added — auto-emit `.dockerignore` baseline

The framework now writes `<repo>/.dockerignore` from a conservative baseline as part of build-context preparation, iff the repo doesn't already have one. User-authored `.dockerignore` is preserved unchanged.

**Implication for the Skill: manifest-declared secret routing is now authoritative for bridge steps.** Pre-0.2.9, a gitignored `.env` containing DB creds could leak into the image via `COPY . .` and be read directly by the container (bypassing `--secrets-from-env`). With 0.2.9's baseline excluding `.env` / `.env.local` / `.env.*.local`, that leak is closed by default.

## Skill behavior on 0.2.9

- **Phase 3 step 7 (`.gitignore` update)**: still required. Keeps the maintainer's git tree clean; orthogonal to Docker context.
- **Phase 3 (new sub-step or addition to step 7)**: if the framework auto-emitted `.dockerignore` (check for the `# Auto-generated by plutus-verify` header), append per-tier exclusions before continuing:
  - **Tier 1 (committed CSVs)**: keep `data/` IN the build context (the script reads from it at runtime). Do NOT add `data/` to `.dockerignore`.
  - **Tier 2 (Drive-backed)**: append `data/` to `.dockerignore` — the container downloads on demand; committed data is not needed in the image.
  - **Tier 3 (DB-backed)**: append `data/cache/` (or equivalent) to `.dockerignore` — prior-run caches must NOT leak into next image (this was the cache-leak incident on cs408-2026/Group09-BuyHighSellLow that motivated 0.2.9).
- **Phase 4 diagnosis**: if a bridge step exits 0 suspiciously fast on first run, suspect cache leak. Inspect with:

  ```bash
  docker run --rm plutus-v2:<hash> ls -la /srv/repo | grep -E "\.env|data/cache|\.git"
  ```

  If `.env` or cache shows up, the framework's `.dockerignore` baseline isn't being applied (older version) or the user's `.dockerignore` is missing those entries. Surface as Phase 4.5 architectural note.

## Known limitation — runtime volume mount bypasses `.dockerignore`

`runner_docker.py` runs each step with `-v {cwd}:/srv/repo`. This is an unfiltered Docker volume mount and overlays the image filesystem with the host's working directory verbatim. `.dockerignore` exclusions apply only to the image build context, not to runtime.

Consequence: even with 0.2.9's `.dockerignore` correctly excluding `.env`, `data/cache/`, etc. from the image, the container still reads them at runtime via the mount. The Tier 3 bridge step can still:

- Pull DB credentials from a host `.env` (bypassing `--secrets-from-env` routing).
- Short-circuit on a host `data/cache/*.parquet` left over from a prior `plutus check` run.

**Phase 4 diagnosis** — if `data_collection` exits 0 suspiciously fast on the second run, compare mtimes pre- and post-check:

```bash
stat -f '%Sm' data/cache/<symbol>/1min/<month>.parquet
```

Unchanged mtime = the bridge short-circuited on the volume mount. 0.2.9 does NOT close this. Until the mount is reworked, either remove the local cache before each `plutus check` (`rm -rf data/cache/`) or wire `--secrets-from-env` properly (real env vars exported in the shell, no host `.env` readable via the mount).

Cross-version posture for the bridge step's "real channel":

| Version | Image-layer leak | Runtime-mount leak |
|---|---|---|
| 0.2.8 and earlier | leaks via `COPY . .` | leaks via mount |
| 0.2.9 | **closed** | still leaks |
| Future | closed | needs mount rework |

See [G11 in known-gotchas](known-gotchas.md#g11) for the operational recipe.

## Backward compatibility

Repos with an existing `.dockerignore` (user-authored or from a prior version of plutus-verify) are unaffected — detection skips the write. Repos without one get the baseline on next `plutus check`; commit or `.gitignore` as preferred.

## What didn't change

- No CLI / SDK / manifest schema changes.
- Existing `.gitignore` semantics unchanged (those are separate concerns).
- `--secrets-from-env` still routes only manifest-declared keys; the new baseline just ensures `.env` doesn't backdoor secrets through the image layer.
