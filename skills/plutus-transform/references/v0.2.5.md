# plutus-verify 0.2.5 notes

Deltas from 0.2.0. When `plutus_verify.__version__` reports `0.2.5`, apply these constraints.

> Source: [docs/others/zbounce-v1-to-v2-upgrade.md](../../../docs/others/zbounce-v1-to-v2-upgrade.md) §6.

## Breaking — manifest

- `expected[].reference_outputs:` renamed to `expected[].artifacts:`. One-liner migration: `sed -i.bak 's/reference_outputs:/artifacts:/g' .plutus/manifest.yaml`.

## Breaking — CLI

- `plutus snapshot --no-reference-outputs` → `plutus snapshot --no-artifacts`.

## Breaking — Python API

- `from plutus_verify.spec import ReferenceOutput` → `Artifact`.
- `ExpectedBlock.reference_outputs` → `.artifacts`.
- `V2RuntimeResult.reference_results` → `.artifact_results`.

## Behavior change

- `visual_similarity` is now opt-in via `plutus check --visual-check`. Without the flag and a configured vision client, comparisons skip non-blockingly. Pre-0.2.5 they hard-failed when no client was configured.
- See **G4** in [`known-gotchas.md`](known-gotchas.md) — 0.2.5's strict missing-`.plutus/expected/` gate still fires *even with* `--visual-check` absent. Workaround: `plutus snapshot --no-run --no-metrics .` to seed expected snapshots, or omit `artifacts:` from the manifest.

## Added

- `unit="fraction"` — new for values in `[0,1]` or `[-1,0]` (drawdowns, returns, HPR). `"ratio"` (the default) still validates for backward compat but should be reserved for unbounded ratios (Sharpe, Sortino).
- `httpx>=0.27` is now a declared dep — pre-0.2.5 wheels imported it without declaring → fresh installs crashed on `plutus --help`.

## Fixed

- `plutus_verify.__version__` was hardcoded `"0.1.0"` while `pyproject.toml` said `0.2.0`. Now consistent.

## Skill behavior on 0.2.5

- Phase 3 manifest: emit `expected[].artifacts:`, not `reference_outputs:`. Use `unit="fraction"` for bounded metrics.
- Phase 3 step 5b chart baseline: still the right approach on 0.2.5 (copy README-referenced charts into `.plutus/expected/<step>/<path>` during manifest authoring). This is the load-bearing fix for the 0.2.5 strict-gate issue described in **G4** — if `.plutus/expected/` is empty when `plutus check` runs, the dispatcher fails the artifact comparison BEFORE the comparator-level skip can fire.
- Phase 4 verification: **do NOT use `plutus snapshot --no-run --no-metrics .` on D4 = path A.** Older versions of this skill ran snapshot in Phase 4 step 2, but timing was wrong — by then the smoke-run had already overwritten the v1-committed chart files, and the snapshotted baseline was tautological. The Phase 3 step 5b copy (before any execution) is the correct mechanism. See **G7** in [`known-gotchas.md`](known-gotchas.md).
- Path B (D4 = script is truth): `plutus snapshot --no-run --no-metrics .` in Phase 4 step 2 is still correct — that path explicitly wants the current run's output as the baseline.
- Phase 4 diagnosis: pre-0.2.6, container stderr is not surfaced in the `plutus check` report. Use the manual `docker run` repro from **G1** / **G5** when a step FAILs with no visible reason.
