# plutus-verify 0.2.8 notes

Deltas from 0.2.7. When `plutus_verify.__version__` reports `0.2.8` or higher, apply these constraints.

> Source: [CHANGELOG.md `[0.2.8]`](../../../CHANGELOG.md).

## Added — `pyproject.toml` as first-class dependency spec

The framework's Dockerfile generator now dispatches on `env.requirements_file`:

| filename | emitted pip command |
|---|---|
| `requirements.txt` (or any non-toml) | `RUN pip install --no-cache-dir -r <file>` |
| `pyproject.toml` | `RUN pip install --no-cache-dir .` |

`plutus bootstrap`'s file detection prefers `pyproject.toml` over `requirements.txt` when both exist.

## Skill behavior on 0.2.8

- **Phase 1 dependency probe**: check `pyproject.toml` first, fall back to `requirements.txt`. Record which one was found; this is the value that goes into `env.requirements_file` in Phase 3.
- **Phase 3 step 2 venv install**:
  - If pyproject.toml present: `pip install .` then `pip install plutus-verify`.
  - Else: `pip install -r requirements.txt` then `pip install plutus-verify`.
- **Phase 3 step 5 manifest authoring**: override the template's hardcoded `requirements_file: requirements.txt` with whatever Phase 1 detected. The three templates ship with `requirements.txt` for historical reasons; pyproject.toml is now the more common case.
- **D5 strip-all**: applies to either file kind. For pyproject.toml, strip version specifiers from the `dependencies` list (and the equivalent for `[tool.poetry.dependencies]` / `[tool.uv.sources]` if present). For requirements.txt, strip from each line as before.
- **G2 diagnosis**: now also covers pyproject.toml resolution-impossible errors. Same fix path (D5 strip-all).

## Backward compatibility

Repos with only `requirements.txt` are unaffected — detection still finds it, framework still emits `pip install -r`. The change is purely additive: a new branch in dockerfile_gen's `if env.requirements_file:` block, and a new priority arm in bootstrap's detection helper.

## Pre-0.2.8 versions

If `plutus_verify.__version__` < `0.2.8` and the repo has only `pyproject.toml`, the Skill should:
- Surface this as a Phase 6 divergence ("pyproject.toml support requires 0.2.8+ — current version is X.Y.Z").
- Either: convert pyproject.toml to a requirements.txt via `pip-compile` (if available) before manifest authoring; or document the limitation in the transform summary and have the maintainer pin to 0.2.8+.
