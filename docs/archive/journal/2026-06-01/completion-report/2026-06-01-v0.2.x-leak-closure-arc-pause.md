## What this arc was about

Over six releases — 0.2.5 through 0.2.10 — the v2 verifier moved from
"works on the upstream `ProtoMarketMaker` reference repo" to "the
manifest's declared posture is actually enforced when run against a
real downstream Tier 3 repo." The path was driven by a feedback loop
with `~/algotrade-research/cs408-2026/Group09-BuyHighSellLow`, a
real DB-backed student strategy: each release, the Skill ran against
that repo and the LLM operator wrote findings to
`<repo>/.plutus/skill-feedback.md`. Those notes were the test-bench
output that drove the next release.

## Leak-channel posture (the load-bearing claim)

After 0.2.10, three previously-open verification-correctness leaks
are closed for Tier 3 bridge steps:

| Channel | ≤ 0.2.8 | 0.2.9 | **0.2.10** |
|---|---|---|---|
| `.env` recoverable from image layers (docker pull leak) | leaks via `COPY . .` | **closed** | closed |
| `.env` readable inside running container | leaks via `COPY . .` | still leaks via mount | **closed** by staging |
| `data/cache/*.parquet` short-circuits bridge step | leaks via `COPY . .` | still leaks via mount | **closed** by staging |

Translation: a Tier 3 step with `network: bridge` and `secrets: [DB_*]`
either really queries the declared DB via routed env vars, or fails
exit=1 with "missing credentials." There is no longer a third path
("appeared to verify, actually short-circuited on host state").

## Skill architecture (paused state)

- `plutus-transform` — four-phase workflow (Survey → Decide →
  Instrument → Verify), auto-chains into `plutus-scoring` at the end.
  Has 12 numbered gotchas (G1-G12) covering the failure modes
  surfaced by the feedback loop, including the v0.2.10-specific
  `step.inputs` allowlist semantics (G12).
- `plutus-scoring` — three-phase standalone skill (Score → Improvement
  paths → Re-run command). Invokable independently on any v2 repo.
- Both skills live under `skills/` in this repo and are installed via
  symlink at `~/.claude/skills/<skill>` → repo path. Edits to the
  source propagate immediately; no re-install needed.

## Parking lot — pick up here when this resumes

Explicit deferred work, ordered by how load-bearing the next decision is:

1. **Inputs-only mount in the runner (or `inputs:` exhaustiveness
   tooling).** v0.2.10's staging dir copies everything not in
   `.dockerignore` when `step.inputs` is empty (the safe default for
   v0.2.9 compatibility). The tighter posture — bind-mount only
   declared inputs read-only — requires `inputs:` to be exhaustive
   across every script's I/O. G12 caught the developer-pain side of
   this: declaring narrow `inputs:` without listing the script binary
   silently breaks the step. Two paths:
   - Static-analysis tooling that introspects each step's command and
     suggests an `inputs:` set (parses `python -m foo.bar`, follows
     imports, lists the source dirs).
   - A `plutus check --strict-inputs` mode that warns when the
     container reads files outside `step.inputs` (requires
     ptrace/strace or LSM hooks; bigger).

2. **"Files left in staging" warning in the check report.** Currently,
   undeclared writes (script wrote a file, manifest doesn't list it
   in `step.outputs`) are silently dropped at extraction time. A
   maintainer who forgot to declare an output will lose data and not
   know why. Adding a line per dropped path in the check-report
   markdown would close this loop.

3. **Staging cache reuse across steps for very large repos.**
   `populate_staging` is `O(repo size)` per step today. Profiling
   deferred until a real repo shows the cost — for the 100-file
   strategy repos this arc targeted, it's well under a second per
   step. If a multi-GB data-science repo lands, revisit.

4. **PyPI publish.** Runbook ready at
   [`docs/runbook/publishing-to-pypi.md`](../runbook/publishing-to-pypi.md);
   pending decision. The wheel + bundled-wheel pattern is stable
   across the arc.

5. **GPU support (`env.base=python-cuda`).** Out of scope across this
   arc; raises `UnsupportedEnvError`. No demand surfaced from the
   feedback loop.

6. **S3 / object-store data source.** Drive backend works for Tier 2;
   S3 was deferred during Phase A and stays deferred.

7. **Deletion of v1 `extract/plan.py` and the legacy LLM pipeline
   branch.** Still living code in the package; v2 native path doesn't
   touch it but it raises questions every time someone opens that
   module.

## Feedback-loop posture for the pause

The downstream test-bench (`Group09-BuyHighSellLow`) has now reported
5 consecutive iterations: v0.2.7, v0.2.8, v0.2.8 cleanroom, v0.2.9,
v0.2.10. Each iteration produced a `.plutus/skill-feedback.md`
identifying one or more upstream defects or doc gaps. The cumulative
shape is: every release closed something real, and the next iteration
surfaced something new — until 0.2.10, where the only remaining gap
was a Skill-doc tightening (G12), not a framework bug. That's the
signal this line of work has reached a reasonable parking point.

When development resumes, the right next step is probably to run the
Skill against a different downstream repo type — Tier 1 (committed
CSVs) or Tier 2 (Drive-backed) — to surface the next class of gaps.
Tier 3 has been the lens for the whole arc; the other tiers haven't
had the same scrutiny.
