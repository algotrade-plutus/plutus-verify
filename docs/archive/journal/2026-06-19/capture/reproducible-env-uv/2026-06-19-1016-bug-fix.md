# Bug Fix — `--secrets-from-env` forwarded the entire host environment (Bug #2)

> Second of two runtime bugs reported from the 9-step dev-tooling session
> (`plutus-verify-uv-runner-bug.md`). Bug #1 (login-shell PATH reset) shipped in
> 0.4.1; this is Bug #2, shipped in 0.4.2.

## Symptom

After the 0.4.1 fix, uv-env steps **still** failed — but only with the
`--secrets-from-env` flag, even for a manifest declaring no secrets (`secrets: []`):

- `plutus check .` → exit 0, `env: reproducible (locked)`, metrics pass.
- `plutus check . --secrets-from-env` → every step exits 1 with
  `ModuleNotFoundError: No module named 'plutus_verify'` (and `numpy`/`pandas`/…).

## Root cause

[`__main__.py:412`](../../../plutus_verify/__main__.py#L412) built
`secrets = dict(os.environ)` for `--secrets-from-env`, and the v2 orchestrator
forwarded that **whole** dict to every step. [`runner_docker.py`](../../../plutus_verify/runner_docker.py)
emits one `-e KEY=VALUE` per entry, so the container received ~50 host vars
including `-e PATH=<macOS host PATH>`. Docker's `-e PATH` **overrides** the image's
`ENV PATH="/opt/venv/bin:$PATH"`, so `python` resolved on the host PATH (whose only
container-existent entry is `/usr/local/bin`) → the system interpreter, which has
neither the locked deps nor the injected SDK. The pip path tolerated it (deps live
in the system python); uv puts everything in `/opt/venv`, which the injected PATH hid.

This was also a reproducibility/leak smell: the maintainer's host env (PATH, HOME,
editor/agent vars) contaminated the "reproducible" container, so two machines
produced different runs — the opposite of the standard's intent. The behavior even
**violated the verifier's own documented contract**
([`manifest_template_todo.py`](../../../plutus_verify/scaffold/manifest_template_todo.py):
"propagates only the declared keys ... undeclared keys are NOT propagated"), and the
v1 path already did it right ([`execute.py:192`](../../../plutus_verify/execute.py#L192):
`{k: secrets[k] for k in alt.needs_secrets if k in secrets}`). The v2 orchestrator
simply never mirrored that filter.

## Fix

New pure helper [`orchestrator._resolve_step_secrets(declared, pool, step_id)`](../../../plutus_verify/spec/runtime/orchestrator.py),
called per step in `run_v2_pipeline`:

```python
{s.key: pool[s.key]
 for s in declared
 if step_id in s.used_by and s.key in pool and s.key not in RESERVED_SECRET_KEYS}
```

The incoming `secrets` dict is now a **candidate pool**; only declared secret keys,
scoped to each secret's `used_by`, are injected. With `secrets: []`, nothing is
injected — so the host `PATH` can never reach the container.

Reserved-key guard (defense-in-depth, from the adversarial review): a manifest
declaring a secret literally named `PATH`/`HOME`/`LD_LIBRARY_PATH`/`PYTHONPATH`/
`VIRTUAL_ENV`/`UV_PROJECT_ENVIRONMENT` would re-open the exact channel. The
validator now **rejects** such a key at check-time (clear message, not a buried
container failure) and the resolver **drops** it anyway. Constant lives at
[`manifest.py` `RESERVED_SECRET_KEYS`](../../../plutus_verify/spec/manifest.py).

Tests: [`tests/unit/test_orchestrator_secrets.py`](../../../tests/unit/test_orchestrator_secrets.py)
(7 tests) — the host-env-leak regression, `used_by` scoping, the pure resolver, and
the reserved-key guard (resolver skip + validator reject). Watched the leak test
fail first (full env reached the runner), then pass. Suite 539 → 546.

Released **0.4.2**. Docs aligned: known-gotchas.md (the gotcha line was actively
wrong), CLI `--help`, `secret-and-leak-hardening.md` (records this as a fourth leak
channel), `build-and-execute.md`, and `manifest_template_todo.py` (it falsely
claimed the validator warns on empty `used_by`).

## Watch-for

- **A declared secret reaches only the steps in its `used_by`.** A secret needed
  by a step omitted from `used_by` is now starved (the step fails) — intended
  least-privilege enforcement, but a manifest-authoring footgun. The 0.4.1
  known-gotchas already tells authors to extend `used_by` to the step that imports
  the data layer.
- **Empty `used_by` ⇒ injected nowhere, silently.** No warning is emitted (the doc
  that claimed one was corrected). A declared-but-unscoped secret will look present
  but never reach a step.
- **Declared-but-missing is a silent skip.** If a declared secret's env var is
  unset, the resolver drops it and the step runs without it (it may then fail
  opaquely). This diverges from the v1 path, which fails the step up front with
  "required secrets missing." Restoring a v2 diagnostic note is a logged follow-up
  (nice-to-have from the adversarial review).
- **`dockerfile_gen` ignores its `secrets` arg today** (`_ = secrets`). If it ever
  starts emitting build-time `ENV`/`ARG` from secrets, that's a NEW leak surface
  that must filter the same way (the leak-completeness lens flagged it).
- The verification confirmed `runner_docker` is the single `-e` chokepoint and the
  v1/`verify` path's pool is a curated `.env` file (never `os.environ`), so no other
  leak path exists.
