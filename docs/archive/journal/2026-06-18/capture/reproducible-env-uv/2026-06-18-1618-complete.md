# Reproducible-Env (uv + lockfile) — Complete

## Summary

Gave plutus-verify a first-class notion of a **reproducibly-locked environment**:
a manifest can declare `env.manager: uv` + a committed `uv.lock`, and the verifier
restores it exactly with `uv sync --frozen` instead of re-resolving dependencies at
build time. A non-locked env is reported `env: NOT reproducible` (warn-only now,
soft-fail later). Shipped as two workstreams — **B1** (verify-side capability) and
**B2** (the `plutus-transform` skill reframed from strip-pins/requirements-first to
uv-lock-first) — released as **0.4.0**.

## Key decisions (synthesis)

1. **uv is the blessed toolchain; lockfile is the reproducibility primitive.**
   - **Context**: `pip install -r requirements.txt` / unpinned `pip install .` re-resolve at
     build time, so the verifier may not reproduce the author's env.
   - **Decision**: bless uv; `pyproject.toml` + committed `uv.lock` restored via `uv sync --frozen`.
   - **Rationale**: best fit for a verifier (exact + hashed lock, pinned Python, deterministic,
     `--frozen` integrity) over pip-tools/poetry; trading research is PyPI-shaped.
   - **Trade-offs**: uv is young → the verifier pins a uv version (`_UV_VERSION`).

2. **Three calibration decisions, all on the recommended option** (user-approved):
   explicit `manager`+`lockfile` fields (not inferred from lockfile presence); Stage-1
   warn-only soft-fail (don't flip the fleet to PARTIAL overnight); verifier blesses one
   pinned uv version.

3. **Phased, non-breaking rollout.** Default stays `pip`; uv is opt-in; non-locked env warns.
   Versioned 0.4.0 (additive feature, not a break).

4. **Flip plutus-transform's D5 from "strip pins" to "lock once + commit."** Stripping pins
   traded reproducibility for buildability — the wrong tradeoff once uv pins everything via
   the lockfile. G2 conflicts now resolved by loosening the one offending constraint and re-locking.

## Challenges & resolutions

### The staging mount would shadow an in-project uv venv
- **Symptom**: a naive `uv sync` creates `.venv` in the project dir.
- **Root cause**: the 0.2.10 runtime bind-mounts the staging copy over `/srv/repo` at exec time,
  which would hide any build-time `.venv` under `/srv/repo` (pip's system site-packages survive
  because they're not under `/srv/repo`).
- **Fix**: `ENV UV_PROJECT_ENVIRONMENT=/opt/venv` + `PATH=/opt/venv/bin:$PATH` so the locked env
  lives outside the mount; `--no-install-project` so non-package repos build.
- **Watch-for**: any future change to the mount target or WORKDIR must keep the venv outside it.

### uv venvs ship without pip
- **Symptom**: `pip install <sdk wheel>` inside a uv venv would fail.
- **Fix**: install the SDK via `uv pip install --python /opt/venv/bin/python <wheel>`.

### Skill prose vs verify-side fidelity (caught by adversarial verification)
- The verification workflow flagged 3 minors: feature doc missing the 0.4.0+ floor (fixed);
  D5 listed 2 options in SKILL.md vs 3 in decision-tree.md (fixed — mention loosen+re-lock);
  `uv sync --frozen` vs `--frozen --no-install-project` (left — faithful abbreviation; the
  local smoke-run venv genuinely uses plain `uv sync`).
