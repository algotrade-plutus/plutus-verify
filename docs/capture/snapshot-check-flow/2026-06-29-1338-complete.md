# Snapshot / Check Flow — Complete

## Summary

Reworked the `plutus snapshot` / `plutus check` data flow into a clean **two-verb, three-store**
model: `snapshot` blesses groundtruth (running in the *container*), `check` is **read-only**
(regenerates into an ephemeral buffer and diffs against the frozen groundtruth, never touching the
working tree). This closed the two dogfood-reported limitations (laptop-vs-container baselines;
verification mutating the tree) without collapsing the deliberately-separate `.plutus/expected/`
groundtruth. Shipped as 0.5.0 (0.5.1 = docs/skills), suite **554 passed**, and the project's dev
workflow migrated to uv in the same arc.

## What was built

### The model — two verbs over three stores
| Store | Role | Written by | Committed |
|---|---|---|---|
| `.plutus/expected/<step>/…` + `manifest.yaml` numbers | groundtruth / database | **`snapshot` only** | yes |
| `.plutus/results/<step>/…` | per-run harvest buffer **+ inter-step bus** | `snapshot` **and** `check` | no — gitignored |
| `result/…` | human-facing view (README) | `snapshot` only | yes |

- **`snapshot` = bless**: run each step in the container → harvest declared outputs to
  `.plutus/results/` → write groundtruth (`.plutus/expected/` files + `manifest.yaml` metric values)
  and the human-facing `result/` view.
- **`check` = verify**: identical pipeline/harvest → diff `.plutus/results/` against the groundtruth;
  read-only w.r.t. the working tree.

### Code (verify side)
- **CLI (L1)** — `__main__.py` `snapshot_cmd`: dropped the `--no-run` hard block; default path builds
  `make_image_builder()` + `DockerRunner()` and calls `scaffold_snapshot(run_check_first=True)`.
  `--no-run` retained as the local-bytes opt-out.
- **Harvest + bus (L2)** — `spec/runtime/staging.py`: `extract_outputs` now harvests declared outputs
  to `.plutus/results/<step>/` (not the working tree); new `stage_prior_results()` injects earlier
  steps' buffered outputs into a step's sandbox (the inter-step bus, filtered by `step.inputs`); new
  `harvest_committed_outputs()` mirrors shipped `artifact_check` outputs into the buffer for a uniform
  compare.
- **Orchestrator** — `_run_step` clears the per-step buffer, injects the bus after `populate_staging`,
  runs the input preflight against the sandbox and the output preflight against `.plutus/results/`;
  `_compare_artifacts` reads `produced_path` from `.plutus/results/<step>/`.
- **Run hygiene** — `check.py` wipes `.plutus/results/` at run start (symmetric to `.plutus/run/`);
  `real_image_builder.py` adds `.plutus/results/` to the `.dockerignore` baseline.
- **Untouched primitives** — `preflight.py` and `artifact_compare.py` needed **zero** changes; the
  read-only refactor was achieved purely by changing the base paths call sites pass in.

### Dev workflow
- Migrated to **uv**: dev/test toolchain → PEP 735 `[dependency-groups].dev`; `[tool.uv]
  default-groups=["dev"]`; `uv.lock` tracked; build backend stays setuptools; consumer extras
  unchanged. Dev flow is now `uv sync && uv run pytest`.

## Key decisions (synthesis)

1. **Keep `.plutus/expected/` as the frozen groundtruth; reject the single-folder model.**
   - **Context:** the dogfood noticed the same artifact bytes committed twice (`result/` *and*
     `.plutus/expected/`) and asked whether committed `result/` could simply *be* the groundtruth.
   - **Decision:** no — keep a separate, **snapshot-only** groundtruth store.
   - **Rationale:** the groundtruth must be written *only* by `snapshot`, because that's what makes
     "blessed" an explicit, detectable act. `result/` is written by ordinary local runs, so it can't
     prove a deliberate blessing. Two failure cases prove it: (a) *never snapshots* — with a
     snapshot-only store, the empty store ⇒ `check` fails "no blessed reference"; with `result/` as
     groundtruth a plain run silently populates it; (b) *forgets to re-snapshot after a code change* —
     a frozen store detects the drift and fails on purpose; `result/` would have already been
     overwritten to match. So the "duplication" is the *price* of an explicit frozen baseline.
   - **Trade-off:** `result/` and `.plutus/expected/` hold the same bytes (git content-addresses
     identical blobs, so the cost is working-tree clutter, not repo bloat).

2. **Decision 1 — inter-step output propagation = thread harvested outputs forward via
   `.plutus/results/`.**
   - **Context:** once `check` can't write `result/`, a later step can't read an earlier step's
     output through the working tree (the old, secret role of the writeback).
   - **Decision:** `stage_prior_results()` remaps `.plutus/results/<step>/<path>` → the next step's
     `staging/<path>`, gated by `step.inputs`.
   - **Rationale:** it's the only option that survives a *first* snapshot — there is no committed
     groundtruth for a derived intermediate yet, so the later step must consume the earlier step's
     freshly harvested output. `snapshot` and `check` use the identical mechanism (verify the way you
     blessed).
   - **Trade-off:** a step with no declared `inputs` receives *all* prior buffers (consistent with
     `populate_staging`'s dockerignore-governs rule); colliding declared output paths → later step
     wins (sorted by step id, not topological). Fine today; revisit on a real collision.

3. **Decision 2 — `snapshot` writes the human-facing `result/`.** Keeps the README's view in a
   conventional location; `check` never touches it (so no churn). The duplication from Decision 1 is
   accepted and git-deduped.

4. **Metrics channel stays in `.plutus/run/results.json`.** The plan briefly implied moving it into
   `.plutus/results/`; only artifact *files* moved. The metric path already works and is orthogonal to
   L1/L2 — moving it would have churned `load_results` / `_compare_metrics` / metric-bless for nothing.

5. **`.plutus/results/` is wiped per run** (per-step clear in `_run_step` + whole-tree wipe in
   `check.py`), symmetric to the `.plutus/run/` wipe, so a stale prior-run artifact can never be
   compared or blessed.

6. **gitignore is the strategy author's job, not the framework's.** The framework manages
   `.dockerignore` (and added `.plutus/results/` to its baseline) but never writes a user `.gitignore`.

7. **Dev workflow → uv** (build backend left as setuptools). uv.lock tracked for reproducible dev/CI;
   consumers resolve published version ranges and are unaffected.

## Challenges & resolutions

### L1 — `snapshot` blessed from the laptop, not the container
- **Problem:** baselines captured from host-disk outputs while `check` byte-compares in the
  container, so a pass required `laptop bytes == container bytes` — fragile for charts/binaries.
- **Root cause:** the CLI hard-blocked the in-container path (`--no-run` forced, `exit 3`);
  `scaffold_snapshot(run_check_first=True)` already worked but the CLI never wired builder/runner.
- **Fix:** wire the builder/runner into `snapshot_cmd` (mirroring `check_cmd`); default to the
  in-container bless. **Unlocks** byte_exact charts/binaries and baseline regeneration from any
  machine/CI.
- **Watch-for:** the `snapshot` CLI still has no `--secrets-from-env` / `--visual-check` (runs with
  `secrets={}`, `vision_client=None`); steps needing secrets to *produce* baselines aren't covered yet.

### L2 — `check` overwrote the working tree (and the writeback was a hidden data bus)
- **Problem:** `extract_outputs` copied declared outputs back over `result/…`, so verification mutated
  tracked files (README charts churned every run) and `check` wasn't read-only.
- **Root cause:** the writeback doubled as the **inter-step data bus** (step N → step N+1 via cwd), so
  it couldn't simply be removed.
- **Fix:** harvest declared outputs to `.plutus/results/<step>/` instead of cwd; reintroduce the bus
  explicitly as `stage_prior_results()` (Key Decision 2); point comparisons at the buffer. `check` is
  now read-only.
- **Watch-for:** output existence is now checked in `.plutus/results/<step>/`, not `repo_path` — a step
  that exits 0 but writes nothing now correctly surfaces a `missing output(s)` error (previously a
  stale committed file could mask it). Stricter and intended.

### Test migration — mock runners had to start producing outputs
- **Challenge:** ~13 tests used mock runners that produced nothing and leaned on pre-staged
  `repo_path` files. Once outputs must be *harvested from the run*, those tests broke.
- **Resolution:** each mock now writes its declared outputs into the staging cwd (as a real container
  does); the 3-step `spec_v2_minimal` e2e fixture now genuinely exercises the inter-step bus
  end-to-end. (TDD throughout, RED→GREEN.)

### Env gap surfaced by the uv migration
- A pre-existing `test_compare_charts` was red without `cairosvg`/`pillow` (scored `partial`, an env
  gap, not a bug). The uv dev group installs them, so it now passes — don't be surprised it was red
  pre-migration.
