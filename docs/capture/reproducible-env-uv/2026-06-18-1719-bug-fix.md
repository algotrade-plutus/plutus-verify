# Bug Fix — uv-locked env steps fail at runtime (runner login shell drops the venv PATH)

> Reported via `plutus-verify-uv-runner-bug.md`, a handoff from the **9-step
> dev-tooling** session (found while migrating its example repo to the uv
> standard). That standalone file is folded into this capture and removed.

## Symptom

A repo with `env.manager: uv` + `env.lockfile: uv.lock` **builds** successfully
(`env: reproducible (locked)`) but **every executing step exits 1**:

```
File "/srv/repo/src/<pkg>/backtesting.py", line 12, in <module>
    import plutus_verify as pv
ModuleNotFoundError: No module named 'plutus_verify'
```

…and likewise `numpy`/`pandas`/etc. The locked deps and the injected SDK are
clearly installed — just not on the python that runs the step. The 9-step
migration was correct per the documented contract but **held**: `plutus check`
couldn't run its steps until this landed.

## Root cause

The two halves of the uv path disagreed on how the venv becomes active:

- [`spec/runtime/dockerfile_gen.py`](../../../plutus_verify/spec/runtime/dockerfile_gen.py)
  installs deps + the SDK into `/opt/venv` (`UV_PROJECT_ENVIRONMENT=/opt/venv`,
  `uv sync --frozen --no-install-project`) and relies on
  `ENV PATH="/opt/venv/bin:$PATH"` to make that venv's python the default.
  (The venv lives at `/opt/venv`, outside `/srv/repo`, because the per-step
  staging copy bind-mounts over `/srv/repo` at run time and would shadow an
  in-project `.venv`.)
- [`runner_docker.py`](../../../plutus_verify/runner_docker.py) ran every step
  with **`bash -lc`** — a *login* shell. On the `python:<ver>-slim` (Debian)
  base, `/etc/profile` **resets `PATH`** to the system default
  (`/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin`), **dropping
  `/opt/venv/bin`**.

So at step-run time `python` resolved to the *system* interpreter
(`/usr/local/bin/python`), which had neither the locked deps nor the SDK. The
legacy **pip** path was unaffected only because it installs into that same
system python, which survives the login-shell reset.

Proof from the report (same image + command, only the shell flag differs):

```
bash -lc 'PYTHONPATH=src python -m <pkg>.backtesting'  → ModuleNotFoundError   (python = /usr/local/bin/python)
bash -c  'PYTHONPATH=src python -m <pkg>.backtesting'  → {"sharpe_ratio": …}   (python = /opt/venv/bin/python ✓)
```

## Fix

[`runner_docker.py`](../../../plutus_verify/runner_docker.py) — one line:
`args += [image, "bash", "-lc", command]` → `"bash", "-c"`. A non-login,
non-interactive shell sources nothing and inherits the container's `ENV PATH`
verbatim — correct for the uv path, unchanged for the pip path. Metrics are
byte-identical, so green checks re-green with no metric change.

This is the **root-cause** fix, not a symptom patch: `dockerfile_gen` and
`runner_docker` must agree on the activation mechanism, and the mechanism is
`ENV PATH`. The documented alternative (write `/etc/profile.d/10-uv-venv.sh`)
would only paper over the runner still discarding `ENV PATH`.

Shipped as **0.4.1**. Accompanying changes:
- `tests/unit/test_runner_docker.py` (new) — asserts the runner uses a non-login
  shell. Watched it fail on `-lc` first, then pass. Closes the exact gap: the uv
  path was *unit-tested only* against the generated Dockerfile **string**, which
  can't see the runner's shell flag.
- `skills/plutus-transform/references/known-gotchas.md` — the 3 manual
  step-repro commands used `bash -lc`; for a uv repo they'd reproduce *this* bug
  while `plutus check` passes, misleading a debugger. Switched to `bash -c` + a note.
- `docs/design/build-and-execute.md` — corrected the documented `docker run`
  command and the uv activation decision to record the non-login-shell requirement.

## Watch-for

- **`dockerfile_gen` and `runner_docker` are a contract pair.** Any future change
  to how the environment is activated (venv location, conda, a different base
  image's profile behavior) must be validated *through the runner*, not just by
  asserting Dockerfile text. A Dockerfile-string test is blind to the shell flag,
  the entrypoint, and the mount.
- **Login shells reset PATH on Debian.** Any `ENV PATH` activation is fragile
  under `bash -lc`. If a step mysteriously can't find an interpreter/deps that are
  "definitely installed," check `which python` under the *exact* shell the runner uses.
- **No integration test builds a `manager: uv` image and runs a step.** The unit
  test guards the shell flag, but the fuller guard the report recommends (build a
  tiny uv repo, run one step that imports a locked dep + `plutus_verify`, assert
  exit 0) is still absent — the integration suite uses fake runners, so it would
  need a docker-gated fixture. Add it if uv adoption widens.
