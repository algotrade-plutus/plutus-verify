# Plan — v0.2.7: artifact baseline, byte fallback, and skill split

> Previous plan in this thread (v0.2.5/v0.2.6 schema polish + packaging
> fixes) shipped as commit `60e4156` and is documented in
> [`docs/completion-report/2026-05-26-schema-polish-and-fixes.md`](../completion-report/2026-05-26-schema-polish-and-fixes.md).
> This plan replaces it for the next change.

## Context

Two related issues surfaced when the user reasoned through what a
v1→v2 transform actually needs from `plutus check`:

**1. Artifact baseline is asymmetric with metric baseline.** In the
`plutus-transform` Skill, Phase 3 step 5 already has Claude (the LLM
operator) read the target repo's README and populate
`expected.metrics[].value` with README-claimed numbers. For artifacts
the Skill instead leans on `plutus snapshot` later in Phase 4 — which
captures the *current* working-tree state of `steps[].outputs`, not the
README-referenced v1 baseline. In practice the smoke-run that runs
before snapshot has already overwritten the v1-committed chart files,
so the eventual `visual_similarity` comparison degenerates to
new-vs-new (`x == x`). The framing should match metrics: Claude reads
the README, identifies which charts it references, and copies those
v1-committed files into `.plutus/expected/<step>/<path>` as part of
manifest authoring. No separate `plutus snapshot` invocation in the
transform path.

**2. `visual_similarity` is useless without an LLM today.** On 0.2.6,
the comparator returns `skipped=True` whenever `vision_client is None`
— informative but not verifying anything. Useful refinement: when an
LLM isn't available, fall back to byte comparison. Byte-identical
files are a real verification (no judgment needed) — though LLM
judgment is still the gold standard when available, because charts
can be semantically equivalent with byte-different metadata
(timestamps, fonts, floating-point jitter). Byte mismatch in the
no-LLM fallback is *probably* divergence but not certain, so it
should surface as a WARNING that doesn't block exit code — visible to
the user, but they can opt into `--visual-check` for a definitive
verdict.

**3. Scoring is generic — split it out of the transform skill.**
Today's `plutus-transform` Skill bundles compliance scoring (Phase 5)
with the transformation work (Phases 1-4). But scoring is useful for
*any* v2-compliant repo, not just freshly-transformed ones: a v2-native
maintainer should be able to ask "what's our compliance score?" without
invoking the transformation playbook. Two-skill separation:

- `plutus-transform` keeps Phases 1-4 (Survey → Decide → Instrument →
  Verify) plus Phase 6 (Consolidate knowledge). Its final hand-off
  step invokes `plutus-scoring` automatically so the operator sees
  the score after a clean transform.
- `plutus-scoring` (new) takes any v2-compliant repo and emits: a
  per-bucket rubric score, ranked improvement paths, and a
  copy-pasteable re-run command for the maintainer.

The split is mechanically cheap (move `compliance-rubric.md` and Phase
5's instructions into a new skill directory) and improves discoverability
("I just want to score my repo" is its own intent, distinct from "I
want to transform an old repo").

This introduces a fourth `CompareResult` state. The matrix becomes:

| `ok` | `skipped` | meaning | marker | exit-code |
|---|---|---|---|---|
| T | F | verified pass | `ok` | 0 |
| T | T | not verified, no evidence of issue (missing ref, missing LLM + byte match path doesn't apply) | `SKIP` | 0 |
| F | T | divergence detected but inconclusive (bytes differ, no LLM to judge) | `WARN` | 0 |
| F | F | verified divergence | `FAIL` | 1 |

The exit-code logic in
[`plutus_verify/scaffold/check.py`](../../plutus_verify/scaffold/check.py)
currently treats any `not r.ok` as a fail. It must change to treat
`not r.ok and not r.skipped` as a fail — i.e., `skipped=True`
overrides `ok=False` and keeps exit code at 0.

Net intent:

- A repo with a v1 baseline and no LLM: deterministic charts get
  real `ok` lines (byte-identical match); non-deterministic charts
  get `WARN` lines visible in the report but don't block exit. The
  user knows immediately whether they need `--visual-check`.
- A repo with an LLM still gets the same LLM-judged comparison as
  today (visual_similarity remains the gold standard when wired).
- The transform Skill stops misusing `plutus snapshot` for the v1
  baseline problem.

## Decision: framework gets a small change; Skill gets a focused extension

### plutus-verify framework — small but touches three files

**1. `_visual_similarity` no-LLM branch** in
[`plutus_verify/spec/runtime/artifact_compare.py`](../../plutus_verify/spec/runtime/artifact_compare.py):

```python
if vision_client is None:
    # Fallback: byte compare. The kind reports what we actually did
    # ("byte_identical") rather than what was declared
    # ("visual_similarity") — the report should reflect execution,
    # not authorial intent. Byte-identical is a real win. Byte
    # mismatch is inconclusive (chart non-determinism is common), so
    # surface as WARN (ok=False, skipped=True): visible but non-
    # blocking. LLM judgment via --visual-check remains the gold
    # standard for ambiguous cases.
    if expected.read_bytes() == produced.read_bytes():
        return CompareResult(
            ok=True,
            kind="byte_identical",
            detail="bytes match (no LLM check needed)",
        )
    return CompareResult(
        ok=False,
        skipped=True,
        kind="byte_identical",
        detail="bytes differ; pass --visual-check for LLM judgment",
    )
```

`byte_identical` is an internal kind — never declarable by users in
the manifest (the schema's `compare` enum stays
`{json_numeric_tolerance, visual_similarity, byte_exact}`). It's a
result-side label that surfaces in the rendered report so the user
can tell at a glance which comparator path actually ran. Distinct
from the user-declarable `byte_exact` (which has strict
exit-1-on-mismatch semantics for deterministic files like CSVs).

Order of checks in `_visual_similarity` stays the same: missing
expected → SKIP (kind=`visual_similarity`); missing produced → FAIL
(kind=`visual_similarity`); LLM unavailable → byte fallback (new,
kind=`byte_identical`); LLM available → LLM judgment
(kind=`visual_similarity`).

**2. Exit-code logic** in
[`plutus_verify/scaffold/check.py`](../../plutus_verify/scaffold/check.py)
(currently line 65-68):

```python
# old
for step_id, refs in runtime.artifact_results.items():
    for r in refs:
        if not r.ok:
            return 1

# new — skipped=True is non-blocking regardless of ok
for step_id, refs in runtime.artifact_results.items():
    for r in refs:
        if not r.ok and not r.skipped:
            return 1
```

This is the load-bearing change for the WARN semantic. Without it,
`ok=False, skipped=True` would still trigger exit=1, defeating the
"visible but non-blocking" intent.

**3. Renderer** in
[`plutus_verify/scaffold/check_report.py::_render_step`](../../plutus_verify/scaffold/check_report.py):

```python
# replace the existing marker selection for artifact lines
if r.skipped:
    marker = "WARN" if not r.ok else "SKIP"
elif r.ok:
    marker = "ok"
else:
    marker = "FAIL"
```

User-visible lines now read (no-LLM case, declared as
`visual_similarity` in the manifest):

```
ok   byte_identical    result/backtest/hpr.svg      [bytes match (no LLM check needed)]
WARN byte_identical    result/backtest/hpr.svg      [bytes differ; pass --visual-check for LLM judgment]
SKIP visual_similarity result/backtest/hpr.svg      [skipped (no reference at …; run plutus snapshot)]
```

And for the LLM-configured case (declared and judged):

```
ok   visual_similarity result/backtest/hpr.svg      [LLM judged: similar]
FAIL visual_similarity result/backtest/hpr.svg      [score=0.42: divergent layout]
```

The `kind` column tells the user which path actually executed. No
`CompareResult` field additions; the existing `(ok, skipped, kind,
detail)` quartet carries the new state.

**Version bump: 0.2.6 → 0.2.7.** This is a behavior change to a
public-facing comparator semantic — what previously always SKIPped can
now report `ok` (when bytes match) or render different detail (when
they differ). Worth its own version line in `CHANGELOG.md` so anyone
relying on "no LLM always SKIPs" can see the change.

### plutus-transform Skill — extend Phase 3 step 5; remove Phase 5; chain to plutus-scoring

**Phase 3 step 5 extension.** Currently reads (paraphrased): "Author
`.plutus/manifest.yaml` from a template; for D4 = path A,
`expected.metrics[].value` comes from README." Extend it so the same
LLM pass that fills metric values also captures artifact baselines:

> For D4 = path A: while filling `expected.artifacts[]` declarations
> from the template, locate each chart's reference in the README
> (markdown image link `![…](path)` or HTML `<img src="path">`). If a
> referenced file exists at `<repo>/<path>`, copy it (mechanical `cp`)
> to `.plutus/expected/<step_id>/<path>`. This captures the
> v1-published chart as the baseline before any new execution
> overwrites it. Symmetric with how metric values are extracted from
> the README in the same step.

Files in the same chart-referenced position but not declared as a
manifest artifact: skip them (not under the verification contract).
Declared artifacts with no README reference: leave the baseline empty;
the comparator's existing missing-expected SKIP path handles it.

**Phase 4 step 2 (snapshot) goes away for path A.** Renumber
subsequent steps. Path B (`script is truth`) keeps the existing
snapshot-after-smoke-run flow because that's exactly what "current
run is the baseline" means — but flag this is now an explicit
conditional on D4.

**Phase 5 leaves entirely.** All of it moves to the new `plutus-scoring`
skill (described below). What stays in `plutus-transform`:

- The "architectural smells we worked around but didn't fix" surfacing

  Phase 6. This is transform-specific context (Phase 1 + Phase 4
  carry-overs) that a standalone scoring run wouldn't have.
- A new final step at the end of the skill (after Phase 6): invoke
  the `plutus-scoring` Skill via the `Skill` tool with the target
  repo path as the argument. This is the "hook" — operator-side

### plutus-scoring Skill — new standalone skill

Purpose: score any v2-compliant repo against the compliance rubric and
emit ranked improvement paths plus a re-run command. Invokable
standalone or chained from `plutus-transform`.

**Frontmatter description:** "Use when scoring a Plutus-v2 compliant
repo against the compliance rubric — recognises phrases like 'score
this repo's plutus compliance', 'what's our plutus score', 'rate this
repo for plutus-verify', 'how plutus-compliant is this'."

**Workflow (3 short phases, no decisions to make):**

1. **Phase 1 — Score**: Apply
   `skills/plutus-scoring/references/compliance-rubric.md`. Emit
   per-bucket score with one-line reasoning (Reproducible 50, Tidy 25,
   Standardized 10, Innovative 15). Total, rounded to 5%.
2. **Phase 2 — Improvement paths**: Concrete, ranked, ≤4 items for
   cheapest paths to push the score higher. Each item maps to a
   specific rubric bucket and explains the delta.
3. **Phase 3 — Re-run command**: Copy-pasteable terminal block the
   maintainer can run any time to re-verify, independent of any Skill
   session. Reads the target repo's `.plutus/manifest.yaml` for
   `secrets[]` and the current branch.

**Pre-flight:**

- Confirm target repo path. Default to CWD; accept argument.
- Confirm `.plutus/manifest.yaml` exists and is loadable. If not, fail
  fast with "this repo is not v2-compliant; run `plutus-transform`
  first or run `plutus init`."
- Probe `plutus_verify.__version__`; load the matching
  `references/v<minor>.md` if present (scoring nuances per version,
  like 0.2.7's byte-fallback affecting "Reproducible" scoring).

**Verification before completion:**

- Score is emitted with per-bucket reasoning and total.
- Improvement paths are concrete (each names a file/line/setting).
- Re-run command block is visible in transcript with concrete secret
  keys / branch filled in.
- (No `plutus check` invocation in plutus-scoring — that's the
  re-run command's job, run by the maintainer.)

**Files to create under `skills/plutus-scoring/`:**

- `SKILL.md` — the skill instructions, structured per the three
  phases above.
- `README.md` — short doc explaining when to use it (mirroring
  plutus-transform's README structure).
- `install.sh` — install script mirroring plutus-transform's.
- `references/compliance-rubric.md` — **moved** from plutus-transform.
  No edits needed to the rubric itself; just relocate.
- `references/v0.2.7.md` (optional, if scoring nuance per version is
  worth documenting) — note that on 0.2.7+, `WARN byte_identical`
  lines still count as "Reproducible passes" for scoring purposes
  (they don't block exit code; they're informational).

## Verification

1. **Unit suite** — `pytest tests/unit/test_runtime_artifact_compare.py
   -v` must pass with the 3 new test cases. Full `pytest` to confirm no
   regressions.
2. **End-to-end transform on ProtoMarketMaker** (or a freshly-cloned v1
   repo on a scratch branch, to avoid this session's commits):
   - Run the Skill. Verify `.plutus/expected/<step>/result/backtest/*.svg`
     contains the v1-committed bytes (not the smoke-run output):
     ```bash
     diff -q .plutus/expected/in_sample_backtest/result/backtest/hpr.svg \
             <(git show HEAD:result/backtest/hpr.svg)
     ```
     Empty output (no difference) = baseline captured correctly.
   - Run `plutus check . --secrets-from-env` (no `--visual-check`,
     no env vars set). Expected: exit=0 with per-artifact lines reading
     `ok byte_identical … [bytes match …]` (deterministic chart) or
     `WARN byte_identical … [bytes differ …]` (chart with timestamp
     / float jitter).
3. **Determinism test** — re-run `plutus check` twice without any code
   change. Both runs should report identical artifact lines. If the
   first run reports `ok byte_identical`, the second should too. If
   the first reports `WARN byte_identical`, that flags chart
   non-determinism in the script itself — useful diagnostic, not a
   regression.
4. **Divergence test** — modify the script to produce a visibly
   different chart (e.g., change a plot title). Re-run without
   `--visual-check`. Expected: `WARN byte_identical … [bytes differ
   …]`, exit=0 (non-blocking, but visible in the report). With
   `--visual-check` and vision env vars set: expected `FAIL
   visual_similarity`, exit=1.
5. **Native-v2 regression** — confirm the change doesn't affect a
   from-scratch v2 repo. The path B branch in the Skill still uses
   snapshot in Phase 4; native-v2 dev workflow untouched. Existing
   passing artifacts on native-v2 repos continue to render `ok` lines
   for byte-identical files.
6. **plutus-scoring standalone** — invoke the new `plutus-scoring`
   skill on a v2-compliant repo without first running
   `plutus-transform`. Should emit the same score + improvement paths
   + re-run command output that the transform-then-score flow
   produces. Verifies the skill split actually achieved independence.
7. **plutus-transform → plutus-scoring chain** — run
   `plutus-transform` on a fresh v1 repo, end-to-end. After Phase 6
   completes, the transcript should automatically include the
   `plutus-scoring` invocation and its output. No operator
   intervention between the two skills.

## Out of scope (intentionally deferred)

- A `plutus snapshot --from-git-head` flag. The LLM-driven copy in
  Phase 3 step 5 reads from the working tree (which is the v1-committed
  state for a freshly-cloned repo with clean tree). The pre-flight
  uncommitted-changes warning catches the failure case. Add the flag
  only if real users hit baseline contamination.
- Per-format byte normalization (strip SVG `<dc:date>` timestamps, etc.)
  to make byte matches more common. The byte fallback's `bytes match`
  path is already a happy path; if it fires often enough to be useful,
  great. If not, users opt into `--visual-check`. Format-specific
  normalization is a separate, larger design.
- A new `byte_soft` compare kind (warn on mismatch, separate from
  `visual_similarity`). The user's framing — fold byte fallback into
  `visual_similarity` — is cleaner. Authors don't need to learn a new
  kind; the framework picks the right behavior based on whether an LLM
  is configured.
