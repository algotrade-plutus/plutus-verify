# v0.2.7 — byte fallback for `visual_similarity` + skill split

Two threads landed together because they share a session and a
release window:

1. A framework refinement that gives `visual_similarity`
   comparisons real verification value without an LLM (byte fallback
   with a WARN state).
2. A skill restructure that pulls scoring out of `plutus-transform`
   into its own `plutus-scoring` skill, with the transform skill
   chaining to it automatically on a clean run.

Designed and executed in the same conversation as the 0.2.6 work
documented at
[2026-05-26-schema-polish-and-fixes.md](2026-05-26-schema-polish-and-fixes.md).

[`docs/plan/2026-05-27-v0.2.7-artifact-baseline-byte-fallback.md`](../plan/2026-05-27-v0.2.7-artifact-baseline-byte-fallback.md).

## Why the byte fallback

The 0.2.6 release fixed the dispatcher-level "missing expected →
FAIL" bug for `visual_similarity` by making it a non-blocking SKIP.
But with no `vision_client` configured (the default state for any
user without an LLM endpoint), the comparator was a complete no-op:
every `visual_similarity` declaration reported a uniform SKIP
regardless of whether the produced and expected files were
byte-identical or wildly different.

The user surfaced two observations during the v1→v2 transform
discussion:

1. If the bytes happen to be identical, that's *real verification*
   — no LLM judgment needed.
2. If the bytes differ, that *might* be meaningful divergence or
   might be benign chart non-determinism (timestamps, font
   rendering, floating-point jitter). Either way the user should
   *see it in the report* so they know whether to opt into
   `--visual-check`.

The fix: when `vision_client is None`, attempt a byte comparison
and pick the result state from a fuller matrix.

## Why the WARN state

Adding a fourth `CompareResult` state was the minimal change that
honestly distinguishes "we verified equality" from "we verified
divergence" from "we couldn't tell." The byte-mismatch case
specifically needed a state where the result is visible in the
report but doesn't block exit code — chart non-determinism is too
common to gate CI on.

The fields already existed (`ok: bool`, `skipped: bool`) — we just
gave `(ok=False, skipped=True)` a meaningful semantic instead of
treating it as nonsensical. Exit-code logic update was load-bearing:
the old `if not r.ok: return 1` had to become `if not r.ok and not
r.skipped: return 1`, otherwise WARN would still flip CI to red.

## Why split the skill

`plutus-transform` Phase 5 was applying a compliance rubric and
emitting a re-run command. Both outputs are useful for *any*
v2-compliant repo: a v2-native maintainer can ask "what's our
plutus score?" without invoking the transformation playbook. The
existing one-skill bundle conflated two intents:

- "Transform my v1-ish repo into something `plutus check` accepts"
- "Score my v2-compliant repo against the rubric"

Splitting into `plutus-transform` (1) and `plutus-scoring` (2):

- Cleaner discoverability — frontmatter triggers fire on the right
  intent.
- Standalone usability — `plutus-scoring` runs on any v2 repo, not
  just freshly-transformed ones.
- Cleaner Phase 5 → its own skill — the rubric, the improvement
  paths, and the re-run command emit together as one cohesive unit.

The "automatic chain" hook is mechanically simple: the last step of
`plutus-transform`'s SKILL.md instructs Claude (the skill operator)
to invoke the `plutus-scoring` Skill via the `Skill` tool. No
programmatic plumbing — just an operator-side instruction.

## What's still latent (suggested follow-ups)

- **Re-test the wheel install cold.** From the 0.2.6 report's
  follow-up list; still applies. `pip install` from a fresh venv,
  confirm `plutus --help` works without extras.
- **Per-format byte normalization** (strip SVG `<dc:date>` timestamps,
  PNG metadata chunks, etc.) to make byte matches more frequent.
  Worth considering only if real usage shows `WARN byte_identical`
  dominating the report on otherwise-deterministic strategies.
- **Standalone `plutus snapshot` review.** With the transform skill
  no longer using snapshot in Phase 4 for path A, the command is
  primarily for native-v2 development. The current implementation
  (`plutus snapshot --no-run --no-metrics`) iterates *all*
  `steps[].outputs`, not just declared `expected.artifacts[]`. Mostly
  fine, but `.plutus/expected/` ends up holding files that aren't
  under any verification contract. A `--artifacts-only` flag would
  scope this; small ergonomic improvement, not blocking.
