## What shipped

Each `plutus check` step runs in a `tempfile.TemporaryDirectory()`
populated from cwd through the `.dockerignore` filter (plus
`step.inputs` if declared). Outputs flow back via `step.outputs`
(plus `.plutus/run/<step>/` always). The orchestrator's `_run_step`
wraps `runner.run(...)` with `populate_staging` and `extract_outputs`
calls in
[`plutus_verify/spec/runtime/staging.py`](../../plutus_verify/spec/runtime/staging.py).

## Why now

Group09's v0.2.9 feedback identified that the runtime volume mount
(`-v {cwd}:/srv/repo`) bypassed `.dockerignore`. Even though 0.2.9
correctly excluded `.env` and `data/cache/` from the **image build
context**, the runtime mount overlaid them anyway. The bridge step
passed with no DB connectivity actually exercised â€” the
verification-correctness claim ("manifest-declared secret routing is
authoritative") was advisory rather than enforced.

## Skill doc tightening (no framework change)

Group09's 5th-run feedback identified that the `step.inputs`
description in `skills/plutus-transform/references/v0.2.10.md` was
misleading â€” read as "the data inputs" rather than "the
complete-coverage allowlist of paths that reach staging." After a
correctly-routed credential run, the verifier reported `FAIL <step>:
exit=2` for every script-executing step because the script binary
itself wasn't listed in `inputs`.

Resolution: the Skill's Phase 3 step 6.5 paragraph and a new
[G12 gotcha](../../skills/plutus-transform/references/known-gotchas.md)
now describe the semantics correctly and recommend `inputs: []` as
the default for new manifests until each step's file surface is
audited. The framework behavior is unchanged â€” `inputs` as
positive-filter is the correct verification primitive; the doc
caught up.

## Known follow-ups (intentionally deferred)

- **Inputs-only mount in the runner itself** (no copy at all,
  bind-mount declared inputs read-only). Tighter semantics; requires
  `step.inputs` to be exhaustive; deferred until a real repo demands
  the stricter posture.
- **"Files left in staging" warning** in the check report. Currently
  undeclared writes are silently dropped â€” visibility would help
  authors notice missing `step.outputs` declarations.
- **Staging cache reuse across steps** for very large repos. Each
  step gets a fresh copy today; profiling deferred until a real repo
  shows the cost.
